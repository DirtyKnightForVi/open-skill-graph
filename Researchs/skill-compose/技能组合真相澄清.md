系统只为选中的技能注册对应的工具函数，**不会**调用任何技能规划器来重新设计技能组合。

## 为什么不用技能规划器？

### 1. **架构设计哲学**
Skill Compose 采用的是**"技能即工具"**的哲学：
- 每个技能都是独立的工具
- LLM 负责决定何时使用哪个工具
- 系统提供工具发现和调用机制

### 2. **简化设计**
避免引入额外的复杂性：
- 不需要技能编排逻辑
- 不需要预定义技能组合模式
- 依赖 LLM 的推理能力进行技能调度

### 3. **灵活性**
让用户的选择直接生效：
- 用户选什么技能，就提供什么能力
- 不强制重新组合或重新设计
- 保持用户意图的原真性

## seed-skills 中的规划类技能的真正用途

通过分析 `skills-planner` 技能，我发现它的真正用途是：

### **辅助设计新智能体**（不是运行时组合）

`skills-planner` 是一个**设计时工具**，用于：
1. **分析用户需求** - 理解用户想要构建什么样的智能体
2. **评估现有技能** - 匹配需求与可用技能
3. **识别技能缺口** - 确定需要创建的新技能
4. **设计智能体架构** - 规划系统提示词和工作流程

### **使用场景**：
- 用户说："我想构建一个客服智能体"
- 系统调用 `skills-planner` 分析需要哪些技能
- 输出推荐技能组合和新技能设计建议
- **但这是设计阶段的辅助，不是运行时的技能组合**

## 实际工作流程对比

### **设计时**（可能用到技能规划器）
```
用户: "我想构建一个XXX智能体"
→ 系统: 调用 skills-planner 分析需求
→ 输出: 推荐技能列表 + 新技能设计
→ 用户: 选择最终技能组合
```

### **运行时**（实际执行，不用规划器）
```
用户: 选择技能 [pdf, imagegen, pubmed]
→ 系统: 构建系统提示词 + 注册工具函数
→ 智能体: 根据提示词和工具执行任务
→ 结果: 各技能协作完成复杂任务
```

## 技术实现的核心区别

### **不是**：技能规划器主动编排
```
❌ 用户选择技能 → 技能规划器重新设计组合 → 生成新工作流程
```

### **而是**：LLM 基于提示词智能调度
```
✅ 用户选择技能 → 系统生成包含所有技能的提示词 → LLM 智能决定使用顺序
```

## 结论

您的观察非常准确！Skill Compose 的技能组合机制确实是：

1. **用户选择技能** → 系统**不干预**用户选择
2. **动态生成提示词** → 告诉智能体有哪些技能可用
3. **LLM 智能调度** → 让 LLM 根据任务需求决定使用哪些技能
4. **工具函数支持** → 提供执行所需的所有工具函数

这种设计更简洁、更灵活，充分信任 LLM 的推理和规划能力，而不是试图用预设的规则来重新编排技能组合。

`skills-planner` 等规划类技能是**设计时辅助工具**，不是**运行时编排机制**，这就是真相！
